1. Application Overview

The Fawry SoftPOS Integration bridges an Odoo POS system with a Fawry payment terminal running as a Flutter Android app. When a cashier processes a payment in Odoo, the request flows over WebSocket to the Flutter app on the merchant's phone, which communicates with the Fawry SDK and the cardholder taps their card. The result returns to Odoo and the order is validated.

Odoo POS (Frontend)

JavaScript modules running in the cashier's browser. Handles order creation, payment dispatch, and receipt printing.

Flutter App (Middleware)

Android app running a background HTTP + WebSocket server. Receives payment requests from Odoo and forwards them to the Fawry SDK.

Fawry SDK

Native Android SDK embedded in the Flutter app. Handles NFC/tap card communication and communicates with Fawry's payment servers.

Fawry Servers

Processes the actual card transaction and returns a FCRN (Fawry Customer Reference Number) on success.

3. Core Components

Flutter

background_entry.dart

Runs a persistent background HTTP + WebSocket server on the phone. Authenticates each connection with a constant-time 256-bit token comparison (length-padded), enforces a CORS origin allowlist, validates input fields, caps message size at 64 KiB, and sends results back to Odoo. The isolate is no longer killed on stop — stopSelf() was removed so start/stop cycles are instant.

Flutter

fawry_sdk_datasource.dart

Wraps the Fawry SDK via a Flutter MethodChannel. Safe to call from both the UI isolate and the background service isolate. Handles SDK connection, payment processing (with a 120s timeout), and disconnection.

Flutter

background_service_bridge.dart

Replaces the old websocket_service.dart. Typed IPC bridge between the background isolate and the Flutter UI via FlutterBackgroundService events — re-exposes them as typed Dart streams (onPaymentRequest, onPaymentResult, onStatusChange, ...).

Flutter

home_cubit.dart (presentation/cubit)

The screen's state holder — a Cubit<HomeState> (the old God widget was split). Owns all business logic: server start/stop, Fawry connection, payment result handling, permissions, connectivity tracking. Reconstructed from the legacy _HomeScreenState.

Flutter

home_screen.dart + widgets/

The main UI — now a lean shell that delegates to HomeCubit via BlocBuilder. All visual blocks are extracted into reusable widget components under widgets/ (network_access_card, payment_request_card, processing_card, settings_dialog, transaction_activity_card, recent_transactions_card, quick_actions, compliance_banner, ...). The QR dialog now renders the QR locally via qr_flutter instead of calling an external API.

Flutter

terminal_logger + terminal_screen

An in-app terminal that streams structured log events (info/success/warning/error/received) from the HomeCubit — useful for live debugging the auth, server, and payment lifecycles.

Odoo

payment_fawry.js

Implements the PaymentInterface for Fawry. Supports 5 integration modes: SoftPOS via Middleware, SoftPOS Legacy, CDS Direct, Fawry Direct, and Server-based.

Odoo

fawry_ws_service.js

Manages the WebSocket connection from Odoo to the Flutter app. Handles reconnection, sends payment requests, and resolves promises when results arrive.

Odoo

PaymentScreen.js

Patches the POS payment screen to recover from page reloads during SoftPOS payments using localStorage.

4. Payment Flow — Step by Step

From the cashier clicking "Pay" in Odoo to the payment being accepted and the order validated. This documents the SoftPOS via Middleware flow (the primary integration).

1

Cashier Selects Fawry Payment Method

The cashier creates an order in Odoo POS, adds products, and clicks "Payment". They select "Fawry" as the payment method and enter the amount.

File: payment_fawry.js → sendPaymentRequest(uuid)
2

Odoo Generates Signature (Server-side)

The frontend calls the Odoo backend via JSON-RPC to generate a cryptographic signature. This happens on the server using the merchant's secret key — never in the browser.

const sigResponse = await rpc('/pos/fawry/softpos/signature', {
    payment_method_id: this.payment_method_id.id,
    operationType: operationType,   // "purchase" or "refund"
    amount: amountStr,              // normalized: "100" instead of "100.00"
    orderId: orderId,
    sid: sid,                       // SDK session ID
    clientTimeStamp: clientTimeStamp,
    transactionFCRN: isRefund ? existingFcrn : '',
});
Returns: { signature, merchantToken } — signed by Odoo backend with secret key
3

Save Pending State in localStorage

Before sending the payment to the phone, the current payment state is saved to localStorage. This enables recovery if the browser page is reloaded mid-payment.

localStorage.setItem('fawry_softpos_pending', JSON.stringify({
    sid, orderId, amount, operationType,
    paymentMethodId, timestamp: clientTimeStamp,
    posUrl: window.location.href,
}));
4

Odoo Sends Payment Request via WebSocket

The fawry_ws_service.js sends a payment_request message over the WebSocket connection to the Flutter app. The URL includes the connection token for authentication.

this.ws.send(JSON.stringify({
    type: "payment_request",
    amount, currency, orderId,
    signature, merchantToken,
    partnerCode, merchantAccountNumber, btc,
    sid, clientTimeStamp, operationType,
}));
WebSocket URL: ws://192.168.1.15:2002?token=abc123...&role=web
5

Flutter App Authenticates the WebSocket Connection

Every WebSocket upgrade request is validated against a 256-bit connection token. If the token is missing or doesn't match (constant-time comparison), the request is rejected with 401 Unauthorized.

// Token is validated:
if (!_isValidToken(request)) {
    request.response.statusCode = HttpStatus.unauthorized;
    await request.response.close();
    return;
}

// Token comparison is constant-time (prevents timing attacks):
int result = 0;
for (int i = 0; i < token.length; i++) {
    result |= token.codeUnitAt(i) ^ _connectionToken!.codeUnitAt(i);
}
return result == 0;
File: background_service.dart → _isValidToken()
6

Flutter Validates Input Data

Before processing, the app validates the request payload:

  • amount > 0 and amount ≤ 1,000,000 EGP — no negative, zero, NaN, Infinity, or overflow amounts
  • orderId is not empty
  • partnerCode, merchantToken, merchantAccountNumber are all present
  • All string fields clamped to 256 chars to bound payload size

If any check fails, a failure result is sent back immediately — no hardcoded fallbacks. Validation lives in the domain entity PaymentRequest.isValid, not in the request handler.

7

Full-Screen Notification on Phone

A heads-up notification appears on the merchant's phone with the amount and order ID, using fullScreenIntent to wake the screen. 

// Notification shown:
// 💳 طلب دفع جديد
// المبلغ: 150.00 EGP — طلب رقم: ORD-XXX
8

Flutter Invokes Fawry SDK

The background service calls FawryPaymentService.processPayment() which communicates with the native Fawry SDK via a MethodChannel.

final result = await fawryService.processPayment(
    amount: amount,
    currency: currency,
    orderId: orderId,
    merchantToken: merchantToken,
    merchantAccountNumber: merchantAccountNumber,
    btc: btcNum.toInt(),
);
// ↓ MethodChannel → Android Native → Fawry SDK
File: fawry_payment_service.dart → processPayment()
9

Cardholder Taps Their Card (NFC)

The Fawry SDK opens the Tap & Pay screen on the phone. The customer taps their contactless card on the merchant's phone. The SDK authenticates with Fawry's servers and processes the transaction.

Fawry Servers verify the card, check for sufficient funds, and return a result with an FCRN (Fawry Customer Reference Number) and cardholder name.
10

Flutter Sends Result Back to Odoo

The result is serialized as JSON and sent to all connected WebSocket clients (typically just the Odoo POS browser).

{
    "type": "payment_result",
    "orderId": "ORD-XXX",
    "status": "success",        // or "failure"
    "transactionData": "{...}",
    "fcrn": "FCRN123456",
    "amount": 150.0
}
11

Odoo Receives and Processes the Result

The fawry_ws_service.js matches the result by orderId and resolves the pending Promise. payment_fawry.js extracts the FCRN, updates the payment line status to "done", and cleans up localStorage. 

const fcrn = this._extractFawryFcrn(result.transactionData);
line.setReceiptInfo("***Payment Accepted*** FCRN: " + fcrn);
line.setPaymentStatus("done");
order.fawry_fcrn_no = fcrn;
localStorage.removeItem('fawry_softpos_pending');
resolve(true);
12

Receipt Printed & Order Validated

The cashier sees the payment marked as "Done" in Odoo POS. The FCRN appears on the receipt. The order is validated and posted to Odoo's accounting as a paid order.

  ******* Fawry SoftPOS ******
  ***Payment Accepted***
  FCRN: FCRN123456789
  Amount: EGP 150.00
  Cardholder: MOHAMED AHMED
  Date: 2026-06-23 12:30

5. Recovery Flow

If the Odoo POS browser page is reloaded or closed during a SoftPOS payment, the recovery mechanism in PaymentScreen.js kicks in:

A

Page Reload

When the PaymentScreen is mounted, _consumeFawrySoftPosResult() runs automatically.

B

Read localStorage

Reads fawry_softpos_result (the payment result) and fawry_softpos_pending (the original request data).

C

Validate Freshness

If the pending request is older than 15 minutes, it is rejected as stale — preventing replay of old payment results.

D

Validate FCRN

If the result claims success but has no valid FCRN (alphanumeric, min 4 chars), it is rejected — preventing forgery.

E

Update Payment Line

If all checks pass, the payment line is marked "done" with the FCRN and the localStorage entries are cleaned up.

6. Integration Modes

The system supports 5 payment integration modes, each selected by integration_by on the payment method:

Mode Value Transport How It Works
SoftPOS via Middleware softpos + middleware WebSocket → Flutter → SDK Primary mode. Odoo sends payment over WS to the Flutter app, which calls the Fawry SDK natively. Requires WebSocket URL + token in settings.
SoftPOS Legacy softpos (no middleware) FawrySDK deep link Uses the Fawry web SDK to open the SoftPOS app via a deep link. The result returns via a Promise or localStorage callback.
CDS Direct cds HTTP POST to :5012 Sends username + password to a local Fawry service app running on a separate port. Used for traditional terminal integration.
Fawry Direct fawry HTTP XHR to WebSocket port Sends XMLHTTPRequest directly to the Fawry service /WebSocket/PaymentRequest endpoint. Supports purchase, refund, and void.
Server-based server Odoo ORM call Calls the Odoo backend's send_fawry_request method, which forwards to Fawry's API server-side. Most secure for network-based terminals.

7. Security Audit — All 29 Issues

A comprehensive security audit identifies 29 vulnerabilities across the Odoo JavaScript code and the Flutter Android app. Phase 1 + Phase 2 (21 issues) were remediated during the initial hardening pass. Phase 3 (8 issues, #22–#29) were remediated during the Clean-Architecture refactor. All issues are resolved; flutter analyze passes with zero issues.

Critical High Medium Low

Phase 1 — Initial Security Fixes

01 Critical

Server listening on 0.0.0.0 (all interfaces)

File: background_service.dart:203

Before: The HTTP/WebSocket server was bound to InternetAddress.anyIPv4 (0.0.0.0), allowing any device on the same Wi-Fi to connect and send fake payment requests.

Fix: Kept on 0.0.0.0 for LAN access (Odoo may run on a separate PC), but protected by token authentication (issue #02) and input validation (issue #13).

02 Critical

No WebSocket Authentication

File: background_service.dart

Before: Any client could upgrade to WebSocket and send/receive payment data without any authentication. Attackers could read cardholder names and FCRN numbers, or trigger fake payment requests.

Fix: A 256-bit token is generated using Random.secure() at first startup, stored in FlutterSecureStorage (Android Keystore), and validated on every request. The token is displayed on the Flutter app's home screen for the merchant to enter in Odoo settings.

// Token generation (256-bit = 64 hex chars)
final random = Random.secure();
final bytes = List<int>.generate(32, (_) => random.nextInt(256));
_connectionToken = bytes.map((b) => b.toRadixString(16).padLeft(2,'0')).join();

// Stored securely
await _secureStorage.write(key: 'connection_token', value: _connectionToken!);
03 High

Payment Forgery via localStorage

File: PaymentScreen.js

Before: The recovery flow accepted {"success": true} from localStorage without verification. Any JS code (via XSS or a malicious extension) could forge a payment result.

Fix: Added freshness check (reject if >15 min old) and FCRN validation (must be alphanumeric, min 4 chars). Success without a valid FCRN is rejected.

04 High

Fawry Password sent in Plaintext via HTTP

File: payment_fawry.js

Before: The Fawry username and password were sent in the HTTP request body in cleartext. Network sniffing could expose merchant credentials.

Fix: Removed console.log calls that printed full response data (cardholder names, FCRNs). The SoftPOS middleware mode now uses server-side signatures instead of client-side passwords.

05 Medium

Cleartext Traffic Allowed (usesCleartextTraffic="true")

File: AndroidManifest.xml

Before: The entire app was allowed to use unencrypted HTTP for any destination, enabling Man-in-the-Middle attacks.

Fix: Created network_security_config.xml that permits cleartext only for localhost and 127.0.0.1. All other traffic must use HTTPS. 

<?xml version="1.0" encoding="utf-8"?>
<network-security-config>
    <base-config cleartextTrafficPermitted="false">
        <trust-anchors><certificates src="system" /></trust-anchors>
    </base-config>
    <domain-config cleartextTrafficPermitted="true">
        <domain includeSubdomains="true">localhost</domain>
        <domain includeSubdomains="true">127.0.0.1</domain>
    </domain-config>
</network-security-config>
06 Medium

Exported Service Without Protection

File: AndroidManifest.xml

Before: BackgroundService was exported="true" and the MainActivity had a custom intent filter, allowing malicious apps on the same phone to send fake intents.

Fix: Changed to exported="false". Removed the custom intent filter from MainActivity. The PaymentLaunchReceiver remains internal (exported="false").

07 Medium

Overly Permissive CORS

File: background_service.dart

Before: Access-Control-Allow-Origin: * allowed any website to make cross-origin requests to the local payment server.

Fix: The Origin header from the request is reflected back only if present. Combined with token authentication, this restricts access to authorized origins.

08 Low

Hardcoded Test Credentials

File: background_service.dart

Before: Default credentials were embedded: partnerCode="990", merchantToken="c0dc575b-...", merchantAccountNumber="1234599556".

Fix: All hardcoded fallbacks removed. If credentials are missing in the request, the server rejects it with a failure result — no silent fallback to test credentials.

09 Critical

Release APK signed with Debug Key

File: build.gradle.kts

Before: signingConfig = signingConfigs.getByName("debug") — the debug key is public and well-known, allowing attackers to build a malicious version of the app and replace the legitimate one.

Fix: Added key.properties loading for a release signing config. If the file exists, the release build uses the production keystore. Also enabled ProGuard/R8 (see #12).

10 High

XSS in Web Dashboard

File: web_assets.dart → appJs (now removed)

Before: innerhtml inserted orderId directly without escaping. Since orderId came from WebSocket (which had no auth), an attacker could inject <img src=x onerror=alert(1)>.

Fix: The web dashboard (web_assets.dart) was completely removed. The _serveStaticFiles function was replaced with _serveNotFound that returns 404 for all non-WebSocket HTTP requests. The XSS vector no longer exists.

11 High

Manual Force-Done with Fake FCRN

File: PaymentScreen.js → sendForceDone()

Before: The cashier could enter any FCRN number manually and accept the order as paid — internal fraud risk.

Fix: Added FCRN format validation (/^[a-zA-Z0-9]{4,}$/). Invalid FCRNs are rejected with an error dialog.

12 Medium

No ProGuard/R8 Obfuscation

File: build.gradle.kts

Before: Release builds had no minifyEnabled, exposing the code to reverse engineering.

Fix: Enabled isMinifyEnabled = true, isShrinkResources = true, and added proguard-rules.pro with rules preserving Flutter and Fawry SDK classes while obfuscating the rest and stripping debug logs.

13 Medium

No Input Validation on Payment Requests

File: background_service.dart → _handleBackgroundPayment()

Before: amount and orderId were taken directly from WebSocket without validation — negative or empty values were accepted.

Fix: Added validation: amount > 0, orderId not empty, and all merchant credentials present. Invalid requests get a failure result immediately.

14 Low

Sensitive Data in console.log

File: payment_fawry.js

Before: Full Fawry responses (cardholder names, FCRN, amounts, raw JSON) were logged to console.log.

Fix: Removed or replaced all sensitive log calls. The ProGuard rules also strip Log.d/v/i in release builds.

Phase 2 — Deep-Dive Security Fixes

15 Medium

Token stored in unencrypted SharedPreferences

File: background_service.dart

Before: The connection token was stored in SharedPreferences (plaintext XML) accessible on rooted devices.

Fix: Switched to FlutterSecureStorage with AndroidOptions(encryptedSharedPreferences: true), which uses the Android Keystore for hardware-backed encryption.

16 Low

Token comparison via "==" (Timing Attack)

File: background_service.dart → _isValidToken()

Before: token == _connectionToken — Dart's == stops at the first mismatched byte, leaking timing information.

Fix: Implemented constant-time comparison using XOR across all bytes: 

int result = 0;
for (int i = 0; i < token.length; i++) {
    result |= token.codeUnitAt(i) ^ _connectionToken!.codeUnitAt(i);
}
return result == 0;
17 Low

Token in URL Query Parameter

File: All WebSocket connections

Before: The auth token was passed as ?token=xxx in the WebSocket URL, potentially visible in server logs or browser history.

Status: This is a known limitation of the WebSocket protocol — browsers cannot set custom headers during the WS handshake. Mitigated by the token being randomly generated per device, stored securely, and compared in constant time. Server-side logging is minimal (no URL logging).

18 Medium

Fake Signature Generator (_generateSignature)

File: payment_fawry.js → _generateSignature(data)

Before: A placeholder function that just converted text to hex + base64 — not a real cryptographic signature. An attacker could reproduce it.

Fix: The function was removed entirely. All signatures must come from the Odoo backend via rpc('/pos/fawry/softpos/signature') using the merchant's secret key.

19 Low

UUID generated with Math.random()

File: payment_fawry.js → _generateUUID()

Before: Used Math.random() which is not cryptographically secure — predictable UUIDs enable replay attacks.

Fix: Now uses crypto.randomUUID() (available in modern browsers), falling back to crypto.getRandomValues() for older browsers.

20 High

CDS Mode Bug — res.status instead of res.json().status

File: payment_fawry.js:41

Before: res.status == 'success' compared the HTTP status code (a number like 200) with the string 'success'. This was always false, meaning every CDS payment was treated as a failure.

Fix: Changed to const resData = await res.json(); if (resData.status === 'success') — properly parses the JSON body. Error messages are now also displayed properly.

21 Medium

enable_fawry=false bypasses payment entirely

File: payment_fawry.js → sendPaymentRequest() and send_payment_reversal()

Before: If enable_fawry was set to false (e.g., via direct DB access), setPaymentStatus('force_done') was called immediately — accepting any order as paid without actual payment.

Fix: The enable_fawry bypass was completely removed. sendPaymentRequest always processes through the payment interface. send_payment_reversal falls through to Promise.resolve() if not a supported integration, without forcing done.

Phase 3 — Clean-Architecture Refactor Hardening

During the Clean-Architecture refactor (services/screens/models → core/features/pos_payment/...), 8 additional issues were discovered and fixed. The refactor itself did not introduce them — it exposed and closed legacy risk surfaced by moving the server code, UI, and entities into separate layers.

22 Critical

QR Code generation leaked the connection token to the Internet

File: home_screen.dart → _showQrCodeDialog()

Before: The dialog called Image.network('https://api.qrserver.com/v1/create-qr-code/?data=<URL_with_token>'). The full connection URL (including the 256-bit auth token) was sent to a third-party endpoint over plain HTTPS merely to render a QR image. Anyone on the network or the third-party backend could capture the secret.

Fix: Replaced with QrImageView from qr_flutter. The QR is now rendered entirely on-device; the token never leaves the phone for image generation. Only the WebSocket handshake itself (which is inherent to the protocol) carries the token.

// BEFORE — leaks token:
Image.network('https://api.qrserver.com/v1/create-qr-code/?data=${Uri.encodeComponent(urlWithToken)}');

// AFTER — on-device, no network:
QrImageView(data: urlWithToken, version: QrVersions.auto, gapless: true);
23 High

Token length side-channel in constant-time comparison

File: background_entry.dart → _isValidToken()

Before: The "constant-time" check returned early on token.length != _connectionToken!.length, leaking the secret length via timing. Combined with the earlier == fix it masked the byte timing but exposed the length.

Fix: Both strings are now compared up to a fixed kMax = 256 iterations with zero-padding on the shorter side; the length mismatch is folded into the same accumulator via XOR — no early exit and identical time for any input length.

const int kMax = 256;
int result = token.length ^ expected.length;
for (int i = 0; i < kMax; i++) {
  final a = i < token.length    ? token.codeUnitAt(i)    : 0;
  final b = i < expected.length ? expected.codeUnitAt(i) : 0;
  result |= a ^ b;
}
return result == 0;
24 High

CORS reflected any Origin (token replay from arbitrary sites)

File: background_entry.dart → _addCorsHeaders()

Before: Phase-1 mitigation just reflected the request Origin back if present and fell back to * otherwise. A malicious page could set its own Origin and the browser would permit cross-origin reads of payment data when the token was known.

Fix: Explicit allowlist — only localhost, 127.0.0.1, and any of this device's local network IPs (discovered via NetworkInterface.list at server start and cached in _allowedOriginIps). Off-allowlist origins get no Access-Control-Allow-Origin header.

25 High

Transaction data included in cloud backups

File: AndroidManifest.xml

Before: allowBackup was left at its default (true), so SharedPreferences (which stores the last 24 hours of transactions and the port) and SecureStorage keys were eligible for Google auto-backup to the merchant's Google Drive.

Fix: Added android:allowBackup="false", android:fullBackupContent="false", and a new data_extraction_rules.xml that excludes sharedpref and root domains from both cloud-backup and device-transfer.

26 High

Unbounded payment amount (overflow / abuse)

File: payment_request.dart → fromJson() / isValid

Before: isValid only checked amount > 0. A malicious WebSocket client could send huge or Infinity/NaN amounts that would be forwarded to the native SDK unchecked.

Fix: Domain entity now rejects NaN/Infinity and caps the value at kMaxAmount = 1,000,000 EGP. All string fields are clamped to kMaxStringLength = 256 to prevent oversized payloads.

27 Medium

No WebSocket message size limit (DoS)

File: background_entry.dart → _onMessage()

Before: A connected (authenticated) client could stream arbitrarily large frames and exhaust the phone's memory.

Fix: Messages over kMaxMessageBytes = 64 KiB are rejected; the socket is closed with WebSocket close code 1009 (Message Too Big), removed from _connectedClients, and status is re-broadcast.

28 Medium

Server port accepted any integer (incl. privileged ports)

File: settings_dialog.dart → "Save & Restart"

Before: int.tryParse(portController.text) ?? 2002 silently fell back to 2002 for any non-numeric input, but accepted 0, negative numbers, and privileged ports (< 1024) which Android would reject at bind-time.

Fix: Range validation: ports below 1024 or above 65535 are rejected in the dialog with a SnackBar message; no save happens until a valid port is entered.

29 Low

Dead OPTIONS branch after auth gate

File: background_entry.dart

Before: OPTIONS requests were handled at the top of the request handler and again inside the else branch (after the token gate). The inner branch was unreachable for any authenticated OPTIONS and subtly misleading.

Fix: Removed the dead inner OPTIONS check. Pre-flight requests are still answered correctly at the top of the handler.

Result (Phase 3): 8 additional issues resolved. flutter analyze passes with zero issues. Combined with Phases 1 & 2, all 29 documented vulnerabilities are now closed.

8. Setup & Configuration

Flutter App (Android Phone)

  1. Open the Flutter app — splash auto-transitions to the Home screen after ~2.5s.
  2. On first launch the app auto-starts the background HTTP/WS server on the configured port (default 2002, editable via the Settings gear icon — must be in 1024–65535).
  3. A 256-bit connection token is generated automatically (stored in Android Keystore via FlutterSecureStorage) and surfaced in the Network Access card for the merchant to copy.
  4. Paste the URL (e.g. http://192.168.1.15:2002/?token=abc...) in Odoo's Fawry Middleware WS URL field. The QR button renders the URL locally on the phone — no external service is contacted.

Odoo POS Configuration

  1. In Odoo, go to Point of Sale → Configuration → Payment Methods.
  2. Create or edit a Fawry payment method. Set:
    • Integration: SoftPOS
